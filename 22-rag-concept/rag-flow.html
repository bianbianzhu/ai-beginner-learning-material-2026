🔎 Lesson 22 · RAG 到底在做什么？

RAG = Retrieval-Augmented Generation（检索增强生成）。 一句话：先查资料，再回答问题。

两条流水线，离线 + 在线

① Ingest Pipeline（离线做，只做一次）

Document PDF / MD / HTML
→
Chunker 800 tokens / 400 overlap
→
Embedder text-embedding-3-small
→
Vector Store 索引好的向量 + 原文

② Query Pipeline（每次用户提问都跑）

Question "配送几次？"
→
Embedder 同一个模型!
→
Retrieve Top-K cosine 相似度
→
Context + Prompt 拼进 instructions
→
LLM → Answer + citations

2026 的 OpenAI 把这一切封装到了什么地方？

流水线环节 你自己写 OpenAI 托管
Chunker str.split("\n\n") 之类 Vector Store 自动切（默认 800/400）
Embedder client.embeddings.create 同上，由 Vector Store 自动调用
Storage + ANN 内存数组 / SQLite / pgvector client.vectorStores.*（前 1 GB 免费）
Retriever cosine + sort + slice(0, k) client.vectorStores.search(...)
LLM 合成答案 自己拼 prompt 调 Responses API tools: [{ type: "file_search" }] 一行搞定

两种方案，各自什么时候用？

🧠 Memory 版（第 24 课）

  • ✅ 零依赖，进程内数组
  • ✅ 本地调试方便，可以打印每一步
  • ✅ 隐私好（不上云）
  • ❌ 进程重启数据就没了
  • ❌ 文件多了索引要跑很久

适合：<100 文档的 demo、单机工具、学习概念。

☁️ OpenAI Vector Store 版（第 25–27 课）

  • ✅ 零运维，OpenAI 负责分块 / 索引 / 检索
  • ✅ 前 1 GB 存储免费
  • ✅ 支持 metadata 过滤、query 改写、ranking 调优
  • ✅ 可以直接接 file_search tool，让 LLM 自己去查
  • ❌ 数据要上云（合规/隐私需要确认）

适合：真实产品、多机部署、需要大规模检索。

RAG 真正的坑在哪里？

📌 记住这 3 件事

  • ingest 和 query 必须用同一个 embedding 模型，否则两边向量空间对不上，检索全是噪声。
  • Top-K 不是越大越好。K=3 常常优于 K=10。多余的 chunk 会稀释相关信号，也浪费 token。
  • 接 API 是最简单的一步。真正难的是：文档怎么切、怎么清洗、怎么评估检索质量、什么问题该拒绝。
