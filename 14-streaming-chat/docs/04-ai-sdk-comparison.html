⚖️ 原生 OpenAI SDK vs Vercel AI SDK v6

你本课手写了 SSE、ReadableStream 拼帧、事件分发、agent loop。
这些事 Vercel AI SDK（2026 主流抽象）全都替你做了。
本页不是推销——是让你看清自己手写了什么，进而决定什么时候值得换抽象。

① 后端：定义一条流式聊天路由

本课的原生实现~80 行

app.post("/api/chat/stream", async (req, res) => {
  const state = getState(req);
  state.chatHistory.push({ role:"user", content: req.body.message });

  res.setHeader("Content-Type", "text/event-stream");
  res.flushHeaders();
  const write = (e, d) => res.write(`event: ${e}\ndata: ${JSON.stringify(d)}\n\n`);

  while (true) {
    const stream = await client.responses.create({
      model: "gpt-5.4-nano",
      input: state.chatHistory,
      tools,
      stream: true,
    });

    let text = ""; let hadTool = false;
    const pending = new Map();
    for await (const ev of stream) {
      switch (ev.type) {
        case "response.output_text.delta":
          text += ev.delta; write("delta", { text: ev.delta }); break;
        case "response.output_item.added":
          if (ev.item.type === "function_call") { hadTool = true; /* ... */ } break;
        case "response.function_call_arguments.delta":
          /* 拼 args buffer */ break;
        case "response.output_item.done":
          if (ev.item.type === "function_call") {
            const r = await runTool(ev.item.name, JSON.parse(ev.item.arguments));
            state.chatHistory.push(ev.item, {
              type:"function_call_output", call_id: ev.item.call_id,
              output: JSON.stringify(r),
            });
          } break;
      }
    }
    if (!hadTool) { state.chatHistory.push({ role:"assistant", content: text }); break; }
  }
  res.end();
});

AI SDK v6 等价实现~15 行

import { streamText, convertToModelMessages, tool } from "ai";
import { openai } from "@ai-sdk/openai";
import { z } from "zod";

app.post("/api/chat", async (req, res) => {
  const result = streamText({
    model: openai("gpt-5.4-nano"),
    messages: convertToModelMessages(req.body.messages),
    tools: {
      get_weather: tool({
        description: "Get weather for a city",
        inputSchema: z.object({ city: z.string() }),
        execute: async ({ city }) => mockWeather(city),
      }),
    },
    stopWhen: stepCountIs(8),   // agent loop 由 SDK 托管
  });

  // UI Message Stream Protocol（SSE 之上的标准化层）
  result.toUIMessageStreamResponse().pipe(res);
});

③ 谁帮你做了什么 —— 对比表

能力 本课原生 AI SDK v6
SSE headers + flushHeaders手写内置
ReadableStream + TextDecoder 拼帧手写（~30 行）内置（UI Message Stream Protocol）
AbortController 传播（client ↔ server）手写内置
Responses API 语义事件分发（text/tool）手写 switch 6 个 typeSDK 抽象掉了
Agent loop（tool call → exec → 再调一次）手写 while 循环内置 stopWhen / ToolLoopAgent
多 provider（OpenAI / Anthropic / Gemini）仅 OpenAI25+ provider，换 2 行代码
UIMessage / ModelMessage 类型安全自己定义 types端到端 typed
AI Elements（预制 React 组件）自己写 UIMessageList/Composer/Reasoning
结构化输出 + 工具调用组合可做内置 stable structured outputs
DevTools（流事件可视化）没有v6 新增
对底层机制的理解强制理解每一层容易"黑盒"

④ 什么时候该用哪个？

场景推荐
教学/学习 / 面试准备 / 想深入理解 LLM 流式原理原生（本课）
Node.js 后端 + 非 Next.js 前端 + 一个 provider 够用都行
Next.js + 多 provider + 想快速上 productionAI SDK v6
特殊流协议需求（需要自己控制 wire format、对接非 HTTP 传输等）原生
需要 UIMessage 持久化/同步到 DB（跨设备聊天）AI SDK v6（UIMessage 就是为此设计）
💡 一句话总结本课的价值：

你现在知道 useChat 是怎么"变"出来的——fetch streaming + TextDecoder + SSE 帧切分 + Responses 语义事件 switch + agent while 循环。

以后用 AI SDK 时，报错你能看懂；性能问题你能定位；特殊需求你能自己补一层。

先懂底层，再用抽象——这才是工程师的 leverage。
