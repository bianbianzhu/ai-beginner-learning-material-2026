🪢 SSE 线格式 + 前端拼帧原理

服务端用什么"格式"把文字流到浏览器？
答案：SSE（Server-Sent Events）——HTTP 的 long-lived 响应，文本协议，Content-Type: text/event-stream，浏览器原生支持。

🧾 SSE 的原始字节长这样

HTTP/1.1 200 OK Content-Type: text/event-stream Cache-Control: no-cache Connection: keep-alive event: delta data: {"text":"你"}\n\n event: delta data: {"text":"好"}\n\n event: delta data: {"text":"，"}\n\n event: tool_start data: {"id":"fc_1","name":"get_weather"}\n\n event: tool_args_delta data: {"id":"fc_1","delta":"{\"ci"}\n\n event: done data: {"usage":{"input_tokens":42,"output_tokens":18}}\n\n

每一"帧"由三部分组成：event: <name>（可选）+ data: <payload> + 一个空行（\n\n）作为分隔符。
空行是关键——它是唯一的帧边界。

🛠️ 前端的拼帧管道

① 网络 chunk
fetch 的 response.body 是 ReadableStream<Uint8Array>。
一次 reader.read() 返回的 chunk 不等于一个完整的 event。
value: Uint8Array([...])
→
② 字符串 buffer
用 TextDecoder 解码。必须 stream: true——避免中文多字节被切在中间变乱码。
buf += decoder.decode(value, { stream: true })
→
③ 按 \n\n 切帧
把 buf 按 \n\n split。最后一段可能不完整，留回 buf 等下一个 chunk。
const frames = buf.split("\n\n");
buf = frames.pop()!;
→
④ 更新 UI state
每一帧解析出 event 和 data，按事件名分发到 setState。
setMessages(prev => [...]);

✅ 一个能跑的最小前端

const resp = await fetch("/api/chat/stream", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ message: text }),
});

const reader = resp.body!.getReader();
const decoder = new TextDecoder();
let buf = "";

while (true) {
  const { value, done } = await reader.read();
  if (done) break;

  buf += decoder.decode(value, { stream: true });   // ← stream:true 必须
  const frames = buf.split("\n\n");
  buf = frames.pop()!;                              // ← 最后一段可能半截，留回

  for (const frame of frames) {
    if (!frame.trim()) continue;
    // 解析 "event: xxx\ndata: {...}"
    const lines = frame.split("\n");
    let ev = "message", data = "";
    for (const l of lines) {
      if (l.startsWith("event: ")) ev = l.slice(7);
      else if (l.startsWith("data: ")) data = l.slice(6);
    }
    const payload = JSON.parse(data);
    handle(ev, payload);     // ← 你的 dispatch
  }
}

⚠️ 4 个容易踩的坑

坑 1：chunk 边界 ≠ 帧边界

一次 chunk 可能拿到 event: delta\ndata: {"text":"你（半截）。直接 JSON.parse 会炸。

✅ 解法：永远 buffer，按 \n\n 切，最后一段留回。

坑 2：TextDecoder 没带 stream:true

中文占 3 字节，一次 chunk 如果切在中文中间，解码会得到 �。

✅ 解法：decoder.decode(value, { stream: true })，让它保留未完成的字节到下次。

坑 3：用 EventSource

EventSource 只支持 GET，不能 POST 带 body。本课用 fetch + ReadableStream。

✅ 2026 的正解：fetch streaming（浏览器 95%+ 支持）。

坑 4：Express 不 flush

默认 response 会缓冲，前端半天看不到字节。

✅ 解法：res.flushHeaders() + 及时 res.write()，不要等整体结束。

💡 本页要带走的 3 条：
  • SSE 的帧边界永远是 空行（\n\n）。你的前端必须 buffer + split + 留尾。
  • 字节 chunk 和 event 帧是两个不同概念。TextDecoder 的 stream:true 是正确解码的关键。
  • 后端要 res.flushHeaders() 并每次 res.write()，否则 TTFT 不存在（全被 Node/Express 缓冲了）。
